# Безопасность

## 1. Главный принцип

Проект не хранит пароль OpenAI.

Он хранит:

- Playwright `storage_state`;
- только в зашифрованном виде;
- локально в вашей базе данных.

## 2. Что теперь защищает доступ к панели

В проект встроена собственная авторизация панели:

- логин происходит через `/api/auth/login`;
- сессия хранится в HTTP-only cookie;
- API и `/evidence` защищены этой сессией;
- операторы панели хранятся в базе;
- пароли операторов хранятся как PBKDF2-хэши, а не в открытом виде.

Bootstrap-переменные в `.env`:

- `AUTH_ENABLED`
- `ADMIN_USERNAME`
- `ADMIN_PASSWORD`

Это не основной каталог пользователей, а механизм первого входа и аварийного восстановления доступа.

## 3. Что надо защищать в первую очередь

### `ENCRYPTION_KEY`

Если получить и базу, и `ENCRYPTION_KEY`, можно расшифровать сохраненные session-state.

Правила:

- не коммитить `.env` в публичный репозиторий;
- хранить резервную копию `.env` отдельно;
- не передавать ключ по небезопасным каналам.

### Каталог `data/`

В `data/` находятся:

- SQLite база;
- evidence;
- зашифрованные session-state.

Это должна быть приватная директория с доступом только у доверенных операторов.

### Доступ к самой панели

Даже со встроенной авторизацией панель не стоит выставлять в открытый интернет без дополнительной защиты.

Рекомендуется:

- запускать в доверенной сети;
- ограничивать доступ firewall-правилами;
- использовать reverse proxy с TLS;
- при необходимости закрывать доступ через VPN.

## 4. Что changed по сравнению со старой схемой

Раньше логин панели был только через `.env`.

Теперь:

- операторы создаются из интерфейса;
- пароль можно менять без редактирования `.env`;
- если все операторы отключены, bootstrap-пользователь из `.env` может восстановить доступ на следующем старте backend.

## 5. Чего не стоит делать

- не хранить `.env` в открытом git;
- не отправлять `storage_state` через мессенджеры и email без шифрования;
- не держать панель публично доступной без TLS и сетевых ограничений;
- не отключать всех операторов сразу, если не хотите уйти в recovery-сценарий;
- не использовать один и тот же простой пароль для bootstrap и рабочих операторов.

## 6. Практические рекомендации

- после первого входа создайте отдельного рабочего оператора;
- bootstrap-пароль из `.env` смените на длинный и не используйте его в ежедневной работе;
- периодически обновляйте session-state для учеток OpenAI;
- делайте резервные копии `data/` и `.env`;
- если подозреваете компрометацию, смените операторские пароли и пересохраните session-state.
