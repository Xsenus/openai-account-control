# Техническое задание  
## Проект: OpenAI Account Control Center

## 1. Основание и цель проекта

Заказчику нужен единый инструмент для контроля нескольких учетных записей OpenAI / ChatGPT, где под одним логином могут одновременно существовать:

- личная подписка (`Free`, `Go`, `Plus`, `Pro`);
- один или несколько `Business` workspace-ов;
- часть команд может быть деактивирована;
- для части workspace-ов у пользователя может не быть полного доступа к billing.

Главная цель проекта — дать оператору **единый центр контроля**, где виден статус всех учеток, всех доступных workspace-ов и признаков остатка / лимита Codex.

## 2. Основные проблемы, которые решает система

Система должна решать следующие практические проблемы:

1. у пользователя много логинов OpenAI;
2. под одним логином могут быть разные workspace-ы;
3. лимиты и usage выражаются не одним числом, а в разных моделях доступа;
4. деактивированные workspace-ы нужно не "терять", а явно помечать;
5. нужно видеть все в одном интерфейсе;
6. нежелательно хранить пароль OpenAI в собственной системе;
7. нужна техническая база, которую можно развивать и поддерживать дальше.

## 3. Область охвата

### 3.1 Входит в scope

- реестр учетных записей OpenAI;
- хранение отдельных browser session-state для каждой учетки;
- ручной и плановый запуск scan;
- обнаружение доступных workspace-ов;
- нормализация результатов в единую модель;
- история scan runs;
- история snapshots;
- React UI;
- FastAPI backend;
- шифрование session-state;
- evidence layer: raw text + screenshot;
- документация и ТЗ;
- helper-скрипты для захвата/импорта session-state.

### 3.2 Не входит в scope

- автоматический обход лимитов;
- распределение рабочих задач между учетками после исчерпания лимита;
- хранение паролей OpenAI;
- перехват скрытых внутренний API OpenAI как обязательный источник данных;
- интеграция с OpenAI Platform billing API для ChatGPT-подписок;
- централизованная SSO/LDAP-аутентификация внутри этого проекта;
- высокая доступность в кластере из нескольких экземпляров.

## 4. Пользовательские роли

### 4.1 Оператор
Человек, который:
- добавляет логины в систему;
- загружает session-state;
- запускает проверку;
- анализирует сводную панель;
- корректирует thresholds.

### 4.2 Технический администратор
Человек, который:
- разворачивает систему;
- генерирует ключ шифрования;
- обслуживает Python runtime и локальные helper-скрипты;
- обновляет parser/selectors при изменении UI ChatGPT;
- анализирует evidence.

## 5. Концептуальная модель предметной области

Система должна различать четыре уровня сущностей:

1. **Account** — конкретный логин OpenAI.
2. **Workspace** — personal или business-контур внутри логина.
3. **Snapshot** — состояние одного workspace на момент времени.
4. **ScanRun** — одна операция проверки одной или всех учеток.

### 5.1 Почему нельзя сводить все к "остатку кодекса"

Потому что с практической точки зрения существуют разные модели:

- included usage;
- credits balance;
- usage-based seat;
- fixed seat + workspace credits;
- частичная видимость данных;
- merged/deactivated состояния.

Следовательно, корректная система должна показывать:
- состояние workspace;
- тип access model;
- текст/тип limit;
- credits;
- общую итоговую оценку риска.

## 6. Функциональные требования

### FR-1. Реестр учеток
Система должна позволять:
- создать карточку учетки;
- хранить label;
- хранить email hint;
- хранить notes;
- включать/выключать участие учетки в scheduler;
- удалять учетку.

### FR-2. Хранение session-state
Система должна:
- принимать `storage_state` Playwright;
- хранить его в зашифрованном виде;
- не хранить пароль OpenAI;
- фиксировать время последней успешной авторизации.

### FR-3. Локальная авторизация через браузер
Система должна поддерживать сценарий:
- пользователь нажимает кнопку в UI;
- backend открывает Chromium через Playwright;
- пользователь входит в ChatGPT вручную;
- session-state сохраняется автоматически.

### FR-4. Импорт session-state JSON
Система должна позволять:
- загрузить session-state вручную из JSON;
- использовать helper-скрипт для захвата сессии;
- привязать полученную сессию к конкретной учетке.

### FR-5. Scan одной учетки
Система должна:
- открывать сохраненную сессию;
- переходить в ChatGPT;
- обнаруживать workspace-ы;
- собирать сигналы по статусу и usage;
- сохранять snapshots.

### FR-6. Полный inventory scan
Система должна:
- запускать проверку всех активных учеток;
- вести общий ScanRun;
- не прерывать batch целиком из-за одной ошибки;
- сохранять статистику по successes/failures.

### FR-7. Планировщик
Система должна:
- запускать полный scan по интервалу;
- позволять менять интервал из UI;
- не требовать внешнего cron по умолчанию.

### FR-8. Нормализация статусов
Система должна рассчитывать общий статус workspace из набора исходных сигналов и поддерживать минимум:
- `OK`;
- `LOW`;
- `BLOCKED`;
- `PARTIAL`;
- `DEACTIVATED`;
- `UNKNOWN`.

### FR-9. История
Система должна сохранять:
- историю scan runs;
- историю snapshots;
- evidence (тексты и скриншоты).

### FR-10. Dashboard
Система должна показывать:
- сколько всего учеток;
- сколько активных;
- сколько с живой сессией;
- число workspaces по статусам;
- последние scan jobs;
- последние нормализованные snapshots.

### FR-11. Настройки
Система должна позволять менять из UI:
- период scheduler;
- порог LOW по credits;
- порог LOW по остатку included usage.

## 7. Нефункциональные требования

### NFR-1. Безопасность
- Пароль OpenAI не хранится.
- Session-state хранится только в зашифрованном виде.
- Evidence хранится локально.
- Система предназначена для приватного self-hosted использования.

### NFR-2. Поддерживаемость
- Модули должны быть разделены на API, services, selectors, models.
- Все функции должны иметь docstring.
- Код должен быть прокомментирован и пригоден для передачи в поддержку.

### NFR-3. Расширяемость
Нужно обеспечить возможность:
- менять selector phrases без переписывания всего проекта;
- добавлять новые parser rules;
- заменить in-memory auth job registry на Redis/DB queue.

### NFR-4. Наблюдаемость
- Каждая проверка должна оставлять след в истории.
- Для спорных случаев должен сохраняться screenshot + raw text.
- Ошибки batch scan должны сохраняться в metrics.

### NFR-5. Простота запуска
- Должен существовать запуск через Python + Node локально.
- Должны быть helper-скрипты и `Makefile`.

## 10. Детализация статусов

### 10.1 `OK`
Присваивается, когда:
- workspace активен;
- сессия валидна;
- credits выше порога;
- остаток included usage выше порога;
- нет явных признаков partial/deactivated/auth_expired.

### 10.2 `LOW`
Присваивается, когда:
- credits меньше либо равны LOW threshold; либо
- процент остатка included usage меньше либо равен LOW threshold.

### 10.3 `BLOCKED`
Присваивается, когда:
- session-state недействителен;
- фактический доступ заблокирован;
- credits исчерпаны и auto top-up отключен.

### 10.4 `PARTIAL`
Присваивается, когда:
- пользователь видит workspace, но не видит полный billing/usage из-за роли или прав.

### 10.5 `DEACTIVATED`
Присваивается, когда:
- workspace отображается как deactivated/locked;
- либо больше не является рабочим состоянием после merge.

## 13. Безопасность и ограничения

### 13.1 Принципы безопасности
- никаких форм для ввода пароля OpenAI в приложении;
- логин выполняется в Chromium/ChatGPT напрямую;
- храним только browser session-state;
- session-state шифруется ключом оператора.

### 13.2 Ограничения
- решение зависит от UI ChatGPT;
- часть информации может быть недоступна для `member` без billing access;
- для headless/remote среды локальный login не подходит — нужен импорт JSON;
- для production нужен контроль доступа к самому приложению на уровне reverse proxy / VPN / локальной сети.

## 14. Тестирование

### 14.1 Минимальный тестовый контур
- unit test: encryption round-trip;
- unit test: status calculation;
- smoke test: создание demo-данных;
- manual test: add account -> import session -> run scan.

### 14.2 Приемо-сдаточные проверки
Система считается принятой, если:
1. запускается локально;
2. позволяет создать учетку;
3. позволяет сохранить session-state;
4. позволяет запустить scan одной учетки;
5. позволяет запустить inventory scan;
6. строит dashboard;
7. пишет историю scan;
8. пишет snapshots;
9. корректно показывает минимум один `deactivated` и один `auth_expired` сценарий.
